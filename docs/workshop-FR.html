2. Introduction à Elm

2.1. Le langage Elm

  • Elm est un langage fonctionnel, dont la compilation génère du Javascript.

  • Il a un typage très fort

Image : utilisateur Henryhartley sur en.wikipedia - licence GFDL ou CC-BY-SA-3.0, via Wikimedia Commons

Elm signifie orme, comme l’arbre, ce n’est pas un acronyme, donc c’est écrit en minuscules.

Elm est l’un des langages fonctionnels les plus simples disponibles aujourd’hui. Cela se traduit par un nombre limité de concepts dans la langue. Cette simplicité est due à la volonté du créateur du langage de le rendre accessible au plus grand nombre.

La notion de type est plus avancée qu’en Java par exemple. Le compilateur est capable de déduire le type de toutes les expressions.

2.2. Un langage front-end

développement en Elm
compilation et optimisation

Elm est un langage front-end uniquement (pour le moment). La compilation produit du code javascript front-end.

Le développeur écrit en Elm, teste en Elm.
Le compilateur Elm compile et optimise le code Elm en un code Javascript concis et efficace.
Ce code n’est pas destiné à être lu par un humain.
Le code Elm n’est pas déployé en production, seul le Javacript compilé l’est.

2.3. Comment l’utiliser ?

  • compiler Elm en JS

  • inclure dans la page HTML le fichier résultant de la compilation

  • appeler la fonction init()

Il est possible pour une application Elm d'interagir avec le code JS natif.

5. Hello, World!

5.1. Lancer un projet Elm

Ouvrez un terminal, et dans le répertoire racine du projet, tapez :

elm init

elm init crée le fichier elm.json qui décrit les dépendances du projet.

5.2. Une application Elm de base

src/Main.elm


module Main exposing (main) -- 

import Html exposing (text) -- 

main =
    text "Hello, World!" --
Un module est déclaré dans un fichier du même nom.
Importe la fonction "texte" du module Html.
La fonction principale appelle la fonction text et lui passe une chaîne de caractères en paramètre

Quelques éléments de syntaxe :

  • Les fonctions et leurs paramètres sont séparés par des espaces et non par des parenthèses.

  • Le nom des types commence par une lettre majuscule.

  • Le nom des fonctions et des valeurs commence par une lettre minuscule.

  • Il n’y a pas de mot-clé return ou équivalent.

  • la valeur retournée est l’évaluation de la dernière ligne exécutée

  • Le résultat de la fonction main est du type Html, et sera rapporté par le runtime elm dans le DOM.

  • L’indentation est significative : le fait que text "Hello, World!" soit indenté implique que cette ligne fait partie du corps de la fonction principale.

  • Le nombre d’espaces utilisés pour l’indentation est libre.

  • -- est le délimiteur pour les commentaires de ligne

Sélectionnez le fichier `elm.json' à la racine de votre projet lorsque l’IDE vous proposera de l’ajouter.

5.2.1. Compiler l’application

Dans le terminal, tapez :

elm make src/Main.elm --output elm.js

  • Le résultat de la compilation est un seul fichier JS, quel que soit le nombre de fichiers source elm.

  • Nous n’éditons jamais ce fichier, dont le contenu hautement optimisé n’est pas conçu pour être lu.

5.3. Formater le code

Si votre IDE/éditeur ne formate pas automatiquement le code, vous devriez le faire régulièrement en tapant dans la console :

 elm-format src/Main.elm  --yes

5.4. Créer index.html

Créez un fichier index.html à la racine du projet, où elm.js a été généré.

index.html :

<!DOCTYPE html>
<html>
<head>
    <meta charset="utf-8" />
    <title>Elm image search</title>
    <script src="elm.js"></script> 
</head>
<body>
    <app></app>

        Elm.Main.init({ node: document.querySelector("app") }) 
    </script>
</body>
</html>
Chargement du fichier résultant de la compilation
Le nœud HTML dans lequel l’application sera affichée
Lance l’application elm

NB : n’oubliez pas de supprimer les légendes ("1", "2", "3") si vous avez copié le code ci-dessus.

La fonction init prend comme paramètre la référence du nœud dans lequel l’application doit être affichée

6.1. Produire du Html

src/Main.elm

module Main exposing (main)

import Html exposing (h1, text)

view =
    h1 [] [text "elm Image Search"] -- 

main =
    view --
appel à la fonction h1 du module Html
appeler la fonction de visualisation

NB : le code ajouté est en caractères gras dans tous les blocs de code.

Le module Html expose des fonctions pour chacune des balises HTML : div, span, p, etc.

Ces fonctions prennent deux paramètres : une liste d’attributs (id, class, onClick, etc.) et une liste d’enfants. Dans l’exemple ci-dessus, h1 n’a pas d’attribut et a un seul enfant, du type texte.

8. Programmation fonctionnelle : les bases de la programmation

Vous pouvez aller directement à Annotations de type si vous êtes déjà familier avec les bases de la programmation fonctionnelle : fonctions pures, fonctions d’ordre supérieur, immutabilité, map/filter/reduce, lambdas…​

8.1. Définition

La programmation fonctionnelle est un paradigme de programmation de type déclaratif qui considère le calcul en tant qu’évaluation de fonctions mathématiques.

Comme le changement d’état et la mutation des données ne peuvent pas être représentés par des évaluations de fonctions la programmation fonctionnelle ne les admet pas, au contraire elle met en avant l’application des fonctions, contrairement au modèle de programmation impérative qui met en avant les changements d’état.

source : Wikipedia

8.2. Immutabilité

Tout en Elm est immuable.

Cela signifie qu’une valeur ou une fonction ne peut pas être modifiée après sa création.

{-- no keyword var, val or def
everything is immutable value, even the functions --}

myNumber = 42

increment x =
    x + 1

Vous ne modifiez jamais la valeur d’une variable, ce n’est pas possible.

Par exemple, l’ajout d’un élément en-tête d’une liste se fait en réutilisant l’ancienne liste. De façon sous-jacent, Le nouvel élément pointe simplement vers le premier élément de l’ancienne liste, qui n’est pas modifiée.

8.3. Fonctions d’ordre supérieur

Les langages fonctionnels permettent l’utilisation de fonctions d’ordre supérieur.

Une fonction est dite d’ordre supérieur lorsqu’elle prend au moins une fonction comme paramètre, et/ou renvoie une fonction comme résultat.

8.4. Fonctions pures

Une fonction est dite pure si elle ne produit aucun effet secondaire.

Le résultat de l’appel à une fonction pure dépend uniquement de la valeur de ses paramètres d’entrée, et ne modifie rien d’autre que sa valeur de retour.

Cela présente des avantages :

  • c’est facile à tester

  • C’est souvent facile à comprendre

  • c’est puissant en cas de multithreading (ce dont Elm ne tire pas parti, car il est basé sur JS)

Toutes les fonctions Elm sont pures. Cela signifie que :

  • elles ne sont basées sur aucun état extérieur

  • elles ne produisent pas d’effets secondaires

  • elles doivent retourner un résultat

C’est pourquoi, par exemple, en Elm, un "if" doit obligatoirement être suivi d’un "else".

8.5. Fonctions de liste

L’immutabilité implique que vous ne pouvez pas faire une boucle "for" dans Elm : vous ne pouvez pas changer la valeur de l’index à chaque itération.

Au lieu de cela, pourquoi vous utiliserez certaines fonctions applicables aux listes, typiques de la programmation fonctionnelle :

  • map pour appliquer une transformation à chaque élément de la liste ;

  • Filtre pour extraire des éléments de la liste correspondant à un critère donné ;

  • Fold (nommé "reduce" dans d’autres langages) pour construire une nouvelle structure de données à partir de la liste.

8.6. Lambda

Lamba = fonction anonyme Lamba = fonction anonyme

Exemple :

List.map (\x -> x + 1) [1, 2, 3]

Un lambda est une fonction qui n’a pas de nom.

Dans l’exemple, la lambda est appliquée à chaque élément de la liste, par la fonction map du module List. Elle ajoutera 1 à chaque élément de la liste. x est la valeur de l’élément courant. Le résultat sera une liste contenant[2, 3, 4].

En Elm, le caractère "antislash" est utilisé pour introduire une fonction anonyme.
Il a été choisi parce que, visuellement, c’est le caractère le plus proche de la lettre grecque lambda.

9. Annotations de type

Les annotations de type sont très utiles en Elm, bien qu’elles soient facultatives. En plus de documenter votre code, elles aident le compilateur à fournir des messages utilisateurs clairs.

L’annotation de type définit le type de retour d’une fonction. C’est écrit juste au-dessus du nom de la fonction, comme :

increment : Int -> Int
increment x =
    x + 1

Une annotation de type en Elm comprend :

  • le nom de la fonction ou de la valeur ;

  • le symbole deux-points ;

  • une série de types séparés par des flèches.

Le dernier type est le type de retour.

incrément : Int → Int signifie que la fonction increment prend un Int comme paramètre et retourne un Int.

Un nom de type dans Elm commence toujours par une lettre majuscule.

sum : Int -> Int -> Int -- takes two Int as parameters, and returns an Int
sum a b =               -- a and b are integers
    a + b

Certains types sont paramétrés. Par l’exemple :

wordCount: List String -> Int -- prend une liste de chaînes de caractères comme paramètre,
                              -- et retourne un Int
wordCount words =
    ...

Un type peut-être générique. Dans ce cas, le nom du type utilisera une ou plusieurs éiquettes en minuscules. Par exemple, List a est une liste générique (similaire à List<Any> ou List<T> dans d’autres langages).

List.length: List a -> Int -- prend une liste de n'importe quel type comme paramètre,
                           -- et renvoie un Int
List.length myList =
    ...

Vous pouvez choisir n’importe quel nom pour les étiquettes, la seule contrainte est de commencer par une lettre minuscule. Il permet de donner une indication de ce que ce type doit contenir.

view: Html msg -- ne prend aucun paramètre,
               -- et renvoie du Html qui peut émettre n'importe quel type de message
               -- ou pas de message du tout.
view =
    text "Hello, World!"

Les parenthèses dans une annotation de type identifient une fonction. Quelques exemples :

apply: String -> (String -> List Int) -> Bool -- prend une chaîne de caractères
                                              -- et une fonction
                                              -- qui convertit une chaîne de caractères
                                              -- en une liste d'entiers,
                                              -- et retourne un booléen

selectTransformation: String -> (Int -> Int -> Int) -- prend une chaîne de caractères
                                                    -- et renvoie une fonction
                                                    -- qui prend deux entiers en paramètres,
                                                    -- et retourne un entier

10. L’architecture Elm

L’architecture Elm est souvent appelée "TEA" (The Elm Architecture)

10.1. Le pattern MVU

Image by Kolja Wilcke

Le principe du pattern MVU est le suivant :

  • l’application comprend un modèle, une fonction de visualisation et une fonction de mise à jour.

  • La fonction de visualisation est en charge du rendu du modèle (elle va générer du Html)

  • La vue émet des messages, par exemple pour indiquer que l’utilisateur a modifié le contenu d’un champ de saisie ou cliqué sur un bouton ;

  • La fonction update traite ces messages et met à jour le modèle en conséquence.

  • Ce modèle s’appelle MVU : Model - View - Update

10.1.1. Le pattern MVU par l’exemple

  1. le modèle contient l’état de l’application (appelé store dans certains frameworks JS). Dans un premier temps il contient : deleteModal=False
    Ce modèle est envoyé à la fonction view par le runtime Elm au démarrage de l’application.

  2. la fonction view utilise le modèle initial pour décider d’afficher ou non la fenêtre de confirmation

  3. l’utilisateur clique sur le bouton "supprimer" de la page html.

  4. la vue émet alors un message indiquant ce clic

  5. le runtime appelle la fonction update, et lui transmet le modèle courant ainsi que le message envoyé par la vue.

  6. la fonction update contient la logique à appliquer pour chaque message

  7. pour DeleteClicked, la logique consiste à stocker dans le modèle la nouvelle valeur du champ deleteModal

  8. à cette fin, la fonction update crée un nouveau modèle, qui est une copie de l’ancien, mais dans lequel elle donne la valeur True au champ deleteModal.

  9. le runtime appelle la fonction view, et lui passe le modèle dont elle doit générer le rendu.

  10. le nouveau modèle contient deleteModal=True, donc la fonction de visualisation affiche la fenêtre modale permettant à l’utilisateur de confirmer la suppression.

10.1.2. Rendons le runtime visible !

10.2. Structure typique d’un fichier Elm

-- import section
import ...

-- main function
main = ...

{-- Type declaration --}
type Model....

{- update function and sub-functions of the update --}
update = ...

{--view function and view sub-functions--}
view = ...

Le MVU est implémenté dans le fichier principal de l’application.
En raison de la nature déclarative du langage, ce n’est pas un problème en Elm de créer un fichier plus long que dans d’autres langages. Ici, les sections sont séparées par des commentaires sur plusieurs lignes.

10.3. Mise en œuvre de l’architecture Elm

TEA uses Browser module

  1. Installer le module Browser

elm install elm/browser
module Main exposing (main)

import Browser
import Html exposing (..)
import Html.exposing attributes (class, type_)

...

Le module Browser expose quatre fonctions qui implémentent l’architecture elm.

Ces fonctions dépendent du type d’application que vous souhaitez créer. La documentation du module Browser les détaille.

10.3.1. Définir un type pour le modèle

Toutes les fonctions du module Navigateur nécessitent un modèle.

Notre modèle sera une simple chaîne de caractère.

import Html.Attributes exposing (class, type_)

type alias Model =
    String

initialModel : Model
initialModel =
    ""

type alias, comme son nom l’indique, vous permet de définir un synonyme pour un type.

initialModel est la valeur - ou fonction, c’est la même chose en Elm, qui définit la valeur initiale du modèle (ici une chaîne vide)

Souvent cette fonction est appelée init, mais pour des raisons de clarté nous l’appellerons initialModel ici.

10.3.2. Invoke Browser.sandbox

Browser.sandbox prend un record comme paramètre

main =
    Browser.sandbox
        { init = initialModel
        , view = view
        , update = update
        }

Un record est le nom en Elm d’une structure.

Le record attendu par la fonction sandbox doit avoir trois champs :

  • init, dont la valeur est celle du modèle initial de l’application.

  • view dont la valeur est la référence de la fonction chargée de rendre visuellement le modèle. Dans notre code, cette fonction est appelée view.

  • update, dont la valeur est la référence de la fonction qui met à jour le modèle. Cette fonction est généralement appelée update.

Il donne les "points d’ancrage" du runtime.

10.4. Définir les événements émis par la vue

Créez un type personnalisé.

import Html.Attributes exposing (class, type_)

type Msg =
    InputChanged String

type alias Model =
    String

Le type Msg est un type personnalisé qui définit les différents messages qui peuvent être envoyés par la vue, et traités par la fonction update.

La déclaration d’un type personnalisé liste les différents constructeurs qui peuvent créer une valeur de ce type. Par exemple, un type booléen serait décrit par : Vrai | Faux

Pour l’instant, il n’y a qu’un seul cas dans lequel un message est envoyé. Nous nommerons le constructeur de type InputChanged, car un message sera envoyé lorsque le contenu du champ de saisie aura été modifié. Comme pour une fonction ou un nom de variable, n’importe quel nom peut être choisi. Il suffit qu’il soit unique et qu’il commence par une majuscule.

Chaque constructeur de type peut accepter 0, 1 ou plusieurs paramètres. Il nous sera utile de connaître le nouveau contenu du champ de saisie. C’est pourquoi nous indiquons que le constructeur InputChanged prend une valeur de type String comme paramètre.

Le résultat d’un appel à InputChanged est la création d’un message de type Msg.

10.4.1. Annotations de type

Ajoutez une annotation de type à la fonction principale.

main: Program () Model Msg
main =

Documentation de Browser.sandbox

  • L’ajout d'annotations de type documente le code et permet au compilateur d’afficher des messages d’erreur plus pertinents.

  • La fonction principale ne prend aucun paramètre, et retourne un Program () Model Msg, puisqu’elle appelle simplement la fonction sandbox du module Browser.

  • Les parenthèses dans l’annotation de type de la fonction principale représentent un tuple vide, appelé unit. C’est une valeur vide.

  • Dans la documentation du module Browser, "msg" commençant par une lettre minuscule n’est pas un type mais une étiquette. Il est équivalent à un type générique (comme <T> dans certaines autres langues). Idem pour le modèle.

10.4.2. Détection d’événements sur la vue

Le module Html.events fournit des fonctions permettant de réagir aux actions de l’utilisateur.

import ...
import Html.Events exposing (onInput)

...

view : Model -> Html Msg -- 
view model =
    div [ class "container" ]
        [ h1 [ class "title"  ] [ text "elm image search" ]
        , input [ type_ "text"
                , class "medium input"
                , onInput InputChanged ] []
        ]
Modifier l’annotation de type pour se conformer aux exigences de Browser.sandbox

Chaque fois que le contenu du champ de saisie est modifié, la vue émettra un message "InputChanged" avec la nouvelle valeur du champ de saisie comme paramètre.

Html est un constructeur de type, qui prend un type de message comme paramètre.

Ce type d’annotation peut être lu comme "La fonction de visualisation du produit du Html qui peut envoyer des messages". Et pas n’importe quel message : seulement ceux décrits par le type Msg que nous avons défini.

10.4.3. Créer une fonction update


view =
    ...

update: Msg -> Model -> Model
update msg model =
    model

La fonction update recevra en premier paramètre un message envoyé par la vue.

update prend comme seconde paramètre le modèle actuel, et retourne le nouveau modèle en résultat.

L’implémentation la plus simple possible pour update est de retourner le modèle actuel, sans le modifier. C’est ce que nous faisons.

Quand on dit que la fonction update met à jour le modèle, c’est par commodité, car le modèle est immuable. En réalité, cette fonction produit un nouveau modèle, éventuellement à partir de l’ancien, chaque fois qu’elle est invoquée.

10.4.4. Mise à jour du modèle

update doit mettre à jour le modèle en fonction du contenu du message. Cette dernière est analysée à l’aide du pattern matching :


update: Msg -> Model -> Model
update msg model =
    case msg of
        InputChanged value ->
            value

Si le message est du type InputChanged, le nouveau modèle sera égal à la valeur transmise par le message (c’est-à-dire le contenu du champ de saisie).

Le pattern matching est une structure de contrôle qui vous permet de choisir un comportement parmi plusieurs, en regardant comment le message a été construit. Il permettra également de déconstruire le contenu du message pour récupérer sa valeur.

Dans notre exemple, msg ne peut avoir été construit qu’en appelant InputChanged suivi d’une chaîne pour le moment.

10.4.5. Débogueur "voyageant dans le temps"

Compilez en mode debug :

10.4.6. Résumons : l’Architecture Elm

L’architecture Elm est la structure des applications dans ce langage. Elle est intégrée au langage, et élimine donc le besoin d’un framework.

  1. le navigateur émet un événement (suite à une action de l’utilisateur par exemple)

  2. le runtime, qui a souscrit à cet événement, invoque la fonction update et lui transmet le modèle courant + un message décrivant l’événement

  3. la fonction update génère un nouveau modèle

  4. le runtime appelle la fonction view et lui passe le nouveau modèle

  5. view génère un nouveau DOM virtuel et le renvoie au runtime

  6. le runtime met à jour le DOM réel

Le code de cette étape est disponible sur le dépôt github, au tag step 3.

11. Communiquer avec un serveur

11.1. Effets de bord

  • Les effets de bord sont gérés par le runtime

  • Les effets de bord sont invoqués via des commandes

  • Le résultat d’un effet de bord est fourni via un message

Nous ne pouvons écrire que des fonctions pures en Elm. Le résultat de l’appel d’une fonction avec des valeurs de paramètres données retournera TOUJOURS la même valeur, ce qui est pratique pour les tests.

Mais que faire si vous devez faire une opération avec un résultat différent ? Par exemple, générer un nombre aléatoire ? Ou faire une demande http ?

Dans ce cas, nous devrons émettre une commande qui demandera au runtime elm pour exécuter la fonction impure. Le runtime fournira le résultat de l’exécution de cette fonction impure. en appelant la fonction update. Le message peut contenir, par exemple, le nombre aléatoire généré.

11.2. Architecture Elm avec effets de bord

Par rapport à la version la plus simple de l’architecture elm, nous aurons des notions supplémentaires pour gérer les traitements qui provoquent des effets de bord.

  1. la fonction update génère un nouveau modèle, et éventuellement une commande

  2. le runtime exécute la commande. C’est là que tous les effets de bord sont traités (requête HTTP, génération de nombres aléatoires, etc.). Les commandes sont asynchrones.

  3. à la fin de l’exécution d’une commande, le runtime appelle la fonction update, et transmet au modèle courant un message qui décrit le résultat de l’exécution de la commande (succès ou échec, valeur, etc) ⇒ retour à l’étape 1

11.2.1. Modifier le type de demande

Remplacez Browser.sandbox par Browser.element puis compilez.

    Browser.element
        { init = init
        , view = view
        , update = update
        }

Browser.element, contrairement à Browser.sandbox, vous permet d’interagir avec le monde extérieur à l’application. Browser.element prend un record comme paramètre avec une zone supplémentaire, les abonnements. Le compilateur indique clairement le champ manquant.

11.9. Digression : le type de résultat

Le type Result représente une réponse à une opération qui peut échouer. Il a deux constructeurs : l’un pour encapsuler le résultat d’une opération réussie, l’autre pour encapsuler une erreur. La définition de ce type dans la bibliothèque de base ressemble à :

type Result error value
    = Ok value
    | Err error

is equivalent to:

type Result a b
    = Ok b
    | Err a

NB : n’ajoutez pas ces déclarations à votre code!

Rappelez-vous que les noms en minuscules ne sont que des étiquettes, pas des types.

11.9.1. Déclarer le type de réponse

La fonction update doit traiter les réponses à la requête HTTP.

type Msg
    = InputChanged String
    | FormSubmitted
    | ResponseReceived (Result Http.Error String)

Le documentation du module HTTP nous indique comment il est utilisé et le format des messages de réponse.

Le message envoyé par le runtime lors de la réception d’une réponse à une requête HTTP inclut un paramètre de type Résultat. Ce type est lui-même paramétré. Le premier paramètre du type Résultat indique le type que le résultat aura en cas d’échec (ici Http.Error). Le deuxième paramètre indique le type du résultat en cas de requête réussie (ici une chaîne de caractères).

(Result Http.Error String) est entre parenthèses car c’est l’ensemble qui forme un type. Sans parenthèses, cela signifierait que le type ResponseReceived a trois paramètres, des types respectifs Result, Http.Error et String - qui ne sont pas compatibles avec le fonctionnement du module Http. Grâce aux parenthèses, nous indiquons que ResponseReceived n’a qu’un seul paramètre, de type Result Http.Error String.

Continuez à suivre les indications du compilateur.

12. Decoder du JSON

12.1. Function Application Operator

Pour convertir le Json en structure de données Elm, nous aurons besoin de l’opérateur |> :

<| signifie : évaluer l’expression à droite, passer sa valeur à gauche
|> signifie : évaluer l’expression à gauche, passer sa valeur à droite

viewUsername : User -> Html Msg
viewUsername user =
    div [] [ text <| user.firstname ++ " " ++ user.name ] --

est équivalent à

viewUsername : User -> Html Msg
viewUsername user =
    div [] [ text (user.firstname ++ " " ++ user.name) ] --

Ces opérateurs permettent une syntaxe très élégante pour chaîner les appels (ex. : appliquer une transformation à tous les éléments, puis filtrer et enfin trier une liste)

12.1.1. Réponse du serveur

La réponse du serveur se présente sous la forme suivante :

{
    "results" : [
        {
            "urls" :
            {
                "thumb" : "http://...",
                "regular" : "http://...",
                ...
            }
        }
    ]
    ...
}

La réponse de l’API à une requête de recherche d’image est un objet JSON qui a une propriété results. La valeur des résultats est un tableau d’objets décrivant des images. Chacun de ces objets possède une propriété url, qui elle-même possède des propriétés thumb et regular qui correspondent aux tailles d’images qui nous intéressent.

12.1.2. Déclarer le type à recevoir

  1. installer le package elm/json

  2. installer le package NoRedInk/elm-json-decode-pipeline

  3. créez un module Image dans un fichier Image.elm avec le contenu suivant :

src/Image.elm

module Image exposing (Image, imageListDecoder)


import Json.Decode as Decode exposing (Decoder, field, int, list, string)
import Json.Decode.Pipeline exposing (required, requiredAt)

type alias Image =
    { thumbnailUrl : String
    , url : String
    }

imageDecoder : Decoder Image -- 
imageDecoder =
    Decode.succeed Image
        |> requiredAt [ "urls", "thumb" ] string -- 
        |> requiredAt [ "urls", "regular" ] string -- 

imageListDecoder : Decoder (List Image) -- 
imageListDecoder =
    field "results" (list imageDecoder)
imageDecoder convertit le JSON reçu du serveur en un type Image.
le contenu de urls.thumb ira dans le 1er champ du record
le contenu de urls.regular ira dans le 2ème champ du record
imageListDecoder transforme le champ "results" du JSON en une liste de records Image,
chacun d’entre eux étant créé à l’aide de imageDecoder.

  • Le premier paramètre de la fonction requiredAt est le chemin de la propriété JSON à récupérer ;

  • le second paramètre, string, est une fonction qui convertit la valeur de la propriété en une valeur String. Notez qu’il s’agit d’un nom en minuscules, donc ce n’est pas le type String.

Il est essentiel dans la déclaration du décodeur de respecter l’ordre des champs du type d’image.

12.1.3. Modifier le type de message reçu

src/Main.elm

type Msg
    = InputChanged String
    | FormSubmitted
    | ResponseReceived (Result Http.Error (List Image))

La réponse sera de type Liste d’images et non plus de type Chaîne de caractères.

  • messages d’erreur utiles

  • compilation rapide

  • cohésion des paquets : pas de conflit entre les paquets

  • performance : il se compare aux frameworks JS les plus rapides, et est meilleur que les plus populaires d’entre eux (source)

  • légèreté des fichiers JS compilés (source)

  • communauté accueillante (essayez #débutant sur l’Elm Slack)

  • Et surtout, un refactoring sans risques, ce qui est probablement sa fonction la plus importante car elle permet de créer des applications qui vieilliront bien.
